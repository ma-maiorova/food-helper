# ADR 001: Архитектура и технологические решения (Food Helper)

**Статус:** принято  
**Дата:** 2025  
**Контекст:** выбор стека и подходов для backend API и dev-окружения проекта Food Helper.

---

## Контекст

Нужно обеспечить backend для поиска готовой еды по КБЖУ и службам доставки: REST API, хранение в БД, стабильный dev-режим и возможность последующего подключения бота и загрузчика данных.

---

## Решения

### 1. Backend: Ktor (Kotlin) вместо Spring Boot

- **Решение:** использовать Ktor 2.x на Kotlin как единственный backend-сервис.
- **Причины:** лёгкий старт, асинхронность из коробки, хорошая поддержка Kotlin, достаточная экосистема (роутинг, плагины, сериализация). Для текущего объёма API Spring был бы избыточен.
- **Следствия:** один модуль `service/`, Netty как движок, конфиг в HOCON (`application.conf`), при необходимости переопределение через переменные окружения (например, БД).

### 2. База данных: PostgreSQL

- **Решение:** реляционная БД PostgreSQL.
- **Причины:** типичный выбор для структурированных данных (продукты, варианты, КБЖУ, связи со службами доставки), расширения (pg_trgm для поиска), хорошая поддержка в экосистеме JVM/Kotlin.
- **Следствия:** схема в виде таблиц `delivery_service`, `product`, `product_variant`; доступ через JDBC (HikariCP) и слой доступа к данным (см. п. 4).

### 3. Миграции схемы: Liquibase

- **Решение:** версионирование схемы БД через Liquibase (YAML changelog в репозитории).
- **Причины:** миграции в коде, воспроизводимость окружений, возможность отката и документирования изменений. Exposed не заменяет миграции на уровне DDL.
- **Следствия:** changelog в `service/src/main/resources/db/changelog/`, запуск миграций при старте приложения после подключения к БД.

### 4. Доступ к данным: Exposed + репозитории

- **Решение:** использовать Exposed для запросов и маппинга в доменные модели; слой репозиториев (интерфейсы + реализации) между API и Exposed.
- **Причины:** типобезопасные запросы на Kotlin, отсутствие тяжёлого ORM, явный контроль SQL и индексов.
- **Следствия:** таблицы описаны в коде (Exposed `Table`), бизнес-логика в сервисах, репозитории изолируют детали БД.

### 5. Единый формат ошибок API (ErrorResponse)

- **Решение:** все ошибки 4xx/5xx возвращать в одном JSON-формате: `requestId`, `code`, `message`, `status`, `path`, `timestamp`, опционально `details` и `fieldErrors`.
- **Причины:** упрощение обработки на клиенте, корреляция с логами по `requestId`, возможность подсветки ошибок по полям через `fieldErrors`.
- **Следствия:** общий DTO `ErrorResponse` (и `FieldError`), StatusPages в Ktor формируют этот DTO; коды ошибок (VALIDATION_ERROR, INTERNAL_ERROR и т.д.) и HTTP-статусы (400, 422, 500) зафиксированы в документации и OpenAPI.

### 6. Идентификация запроса (requestId)

- **Решение:** каждый HTTP-запрос получает идентификатор (requestId): из заголовка `X-Request-Id` или сгенерированный UUID; сохраняется в атрибутах вызова и в MDC для логов.
- **Причины:** трассировка запроса в логах и в ответе об ошибке без привлечения тяжёлых систем.
- **Следствия:** плагин/перехватчик в начале пайплайна устанавливает requestId; формат логов включает requestId; при ошибке тот же requestId возвращается в теле ответа.

### 7. Документация API: OpenAPI 3.0 + Swagger UI

- **Решение:** описание API в OpenAPI 3.0 (YAML), раздача спецификации и Swagger UI средствами приложения (статический YAML + webjars для Swagger UI).
- **Причины:** единый источник правды для контракта API, удобство тестирования и онбординга; независимость от облачных генераторов.
- **Следствия:** файл `openapi/openapi.yaml` в ресурсах, ручки `/openapi/openapi.yaml`, `/swagger-ui/`, редирект `/swagger`; в спецификации описаны успешные ответы и типовые ошибки (400, 422, 500).

### 8. Dev-окружение: Docker Compose + .env

- **Решение:** стабильный dev-режим через Docker Compose: сервисы postgres и api, healthchecks, конфигурация через `.env` (по образцу `.env.example`).
- **Причины:** одинаковый способ поднять окружение у всех разработчиков, возможность позже добавить loader и другие сервисы.
- **Следствия:** `docker-compose.yml` в `service/`, переменные БД и портов в `.env`; API при старте читает, при наличии, `DB_JDBC_URL`, `DB_USER`, `DB_PASSWORD` из окружения.

### 9. Индексы БД под типичные запросы

- **Решение:**  
  - по доставке: индекс по `product.delivery_service_id` (уже в первой миграции);  
  - по поиску названия: расширение pg_trgm, GIN-индекс по `lower(name)` для подстрокового поиска;  
  - по КБЖУ: составные индексы для частых комбинаций (например, `product_variant(product_id, calories)`, `(calories, protein)`, `(calories, fat)`).
- **Причины:** ускорение фильтрации по службам доставки, по тексту и по диапазонам КБЖУ без перебора больших объёмов данных.
- **Следствия:** отдельная миграция (003) с расширением pg_trgm и созданием указанных индексов; при добавлении новых частых фильтров можно вводить новые индексы через changelog.

## Открытые вопросы / будущее

- Выбор стратегии загрузки данных (loader): периодические задачи, очереди, формат источников.
- Интеграция с Telegram-ботом (отдельный сервис или библиотека), протокол обмена с API.
- Аутентификация и авторизация API (если понадобятся закрытые ручки).
- Продакшен-деплой: образы, оркестрация, секреты, мониторинг.
